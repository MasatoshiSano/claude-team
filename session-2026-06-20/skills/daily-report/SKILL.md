# 日報生成 → TaskFlow記録スキル

## このスキルの目的

メンバーが「思い出しながら入力する」のをやめ、**その日の作業の実体から、正確な業務報告を生成して TaskFlow に残す**こと。
入力作業をゼロに近づけ、記録が自然に貯まる状態をつくる。

重要な前提:
- このスキルは「考えて・まとめる」担当。TaskFlow は「貯める」担当。役割を混ぜない。
- 新しいアプリは作らない。器は既存の TaskFlow、頭脳は Claude。

## 起動したら、まず状況を確認する

ユーザーが日報を求めたら、いきなり質問攻めにせず、**まず会話の文脈から作業実体を拾えるか確認する**。

1. **同じ Claude セッションで作業していた場合**(Claude Code 等で開発・作業していた)
   → そのセッションのやり取りから、触れたタスク・実施内容・おおよその所要時間を**自分で推定して下書きする**。ゼロから聞かない。
   → 下書きを見せて「この理解で合っていますか?抜け・修正があれば教えてください」と確認する。

2. **作業実体が会話にない場合**(別環境で作業していた／コーディング以外の業務)
   → 下の「最小ヒアリング」だけ行う。質問は一度にまとめて、3項目まで。

## 最小ヒアリング(実体が拾えないときだけ)

以下を**一度に**尋ねる。埋まったものだけでよい。

- 今日進めた TaskFlow のタスク(またはタスク名・案件名)
- 実際にやったこと(箇条書きで可)
- 詰まった点・明日に持ち越す点(あれば)

所要時間は「ざっくりで可」とし、不明なら無理に聞かない。正確さより継続性を優先する。

## 出力する日報の構造

以下の固定フォーマットで生成する。TaskFlow の業務報告に流し込むことを前提に、機械可読な形も併記する。

### 人が読む日報(本文)

```
■ 日報 YYYY-MM-DD ／ 担当: <名前>

【今日やったこと】
- <タスク名>: <実施内容>(所要: <時間 or 不明>)
- ...

【進捗・状態】
- <タスク名>: <未着手 / 進行中 / レビュー待ち / 完了>

【詰まった点・要相談】
- <あれば。なければ「特になし」>

【明日の予定】
- <あれば>
```

### TaskFlow 記録用データ(構造化)

会話の最後に、TaskFlow へ書き込むための構造化データを **JSON** で出す。
TaskFlow のタスク単位に分解し、1タスク=1レコードにする。

```json
{
  "date": "YYYY-MM-DD",
  "author": "<名前>",
  "entries": [
    {
      "task_id": "<分かれば TaskFlow のタスクID。不明なら null>",
      "task_name": "<タスク名>",
      "work_done": "<実施内容>",
      "duration_min": <所要分。不明なら null>,
      "status": "<not_started | in_progress | review | done>",
      "blocker": "<詰まり。なければ null>"
    }
  ]
}
```

## TaskFlow への記録(2モード)

環境に応じて自動で振り分ける。

### モードA:TaskFlow-MCP が接続されている場合
- 上記 JSON を使い、TaskFlow-MCP のツールで各 entry を該当タスクの業務報告として書き込む。
- task_id が null のものは、TaskFlow 側でタスク名を検索して照合を試みる。照合できなければユーザーに確認する。
- 書き込み後、「TaskFlow に N件 記録しました」と結果を返す。**勝手に既存データを上書きしない。追記する。**

### モードB:MCP が未接続の場合
- 「TaskFlow との自動連携は未接続です」と伝える。
- 人が読む日報(本文)と、コピペ用の JSON の両方を提示し、「TaskFlow の業務報告欄に貼り付けてください」と案内する。
- 自動化したい場合は管理者(チームリーダー)に TaskFlow-MCP 接続を相談するよう促す。

## やってはいけないこと(Anti-patterns)

- **作業を捏造しない。** 会話に実体がなく、ヒアリングでも埋まらない項目は「不明」のままにする。それらしい内容を埋めない。
- **所要時間を勝手に断定しない。** 推定なら「(推定)」と明示する。
- **評価・査定をしない。** 日報は記録であって人事評価ではない。「効率が悪い」等の判断を書かない。
- **長文にしない。** 日報は3分で書ける軽さを保つ。冗長な説明文を足さない。
- **TaskFlow の既存データを上書きしない。** 必ず追記。
- **個人の負荷や遅れを、本人を責める形で書かない。** 事実(状態・ブロッカー)だけを淡々と残す。

## チームリーダー向けの補足(俯瞰での再利用)

このスキルで TaskFlow に記録が貯まると、リーダーは別途
「TaskFlow の今週の業務報告を読んで、遅れている案件・特定メンバーへの偏りを整理して」
と頼むことで、聞き回らずにチーム状況を俯瞰できる。
日報生成(個人が楽になる)と俯瞰(リーダーが楽になる)は、同じ記録から両立する。
