## 1. ユーザーの業務プロフィール（確定情報）

### 役割・立場
- 三菱電機 ソ企。ソフト開発と製造現場の橋渡し。
- プレイングマネージャー。6〜10人のチームを率いる。

### 時間配分（自己申告）
- コーディング 6割 / 会議 3割 / チームマネジメント 1割（誰が何をするか・方向性）

### コーディング業務の内訳
- 新規開発 / 既存改修・機能追加 / バグ対応・保守 / 現場の細かい調整、すべて担当。
- **一番しんどいのは「既存改修・機能追加」と「バグ対応・保守」**。
- しんどさの内訳: 量が多い × 1件が重い × 自分にしか振れない（属人化）。

### 属人化の原因（本人談）
- 仕様が頭の中にしかない（ドキュメント化されていない）
- 振るより自分でやったほうが早い
- 振るための時間・手順を整える余裕がない
→ これらは一本の鎖（ドキュメントがない→説明に時間→自分で抱える→書く余裕なし→ドキュメントがないまま）。

## 2. 依頼〜開発〜展開のフロー（現状の実態）

### 入口（要望の届き方）
- Teams チャット（多くの要望、流れて消える）
- 一部アプリは Issue 入力をお願いしている（ただし情報が薄く、結局ヒアリングし直し）
- **2つの穴**: Teams=記録に残らない / Issue=情報が薄い

### 判断（対応すべきか）
- 「対応すべきか」「個人の意見をどこまで吸い上げるか」を毎回自分が判断。
- 判断軸は明文化されていない（頭の中）→ 人に振れない核心。

### 要件化
- 特に固めず作りながら考える / 頭の中で整理してすぐコード / 簡単なメモから着手。
- **要件化のステップが頭の中で完結し、文書が生まれない**。

### 開発・保守中のやり取り
- 追加・変更要望が途中で出る / 動作確認を現場に依頼 / データ仕様確認 / 仕様の認識合わせ。
- → 最初に仕様を固めていれば減るやり取りが、開発中の手戻りとして噴出。

### 委譲
- **ほぼ振れていない（自分で全部抱えている）**。
- 任せ方はメンバーによる（レベル差。完全に任せられる人/要所で口を出す人）。

### 展開・説明
- スライド作成 / 実際に動かしてデモ / 口頭・Teams。
- **マニュアル・手順書は作っていない**（残る形のドキュメントがない）。
- 「説明したのに使ってもらえない/展開が進まない」ことが時々ある。

### 報告
- 課長へは「聞かれたときだけ」「Teams で随時」。定期報告なし。
- 報告作成の手間は感じていない（＝報告していない）。
- → **自分の負荷が上位層に構造的に見えていない**。

### ドキュメント
- README 程度は作ることがある。
- 仕様書・設計メモは「作りたいが時間がなくて作れていない」（意欲はある、時間がボトルネック）。

### 言語・展開意向
- 成果物は日本語のみでよい。
- product-management は最終的に**チームに広げたい**。

### 半年後の理想（個人）
- ドキュメントが残る仕組み / チームに振れる / 要望の取捨・優先度づけが楽になる。

## 3. チームマネジメントの実態（確定情報）

- 把握すべきメンバー: **6〜10人**。
- メンバーの働き方は**バラバラ**:
  - ツール開発（レベル差あり）
  - 他部署への AI 勉強会・ツール展開
  - ヘルプデスク対応（既存アプリ修正・機能追加・不具合）
  - 情シスとのネットインフラ展開調整
  - 社外へのシステム展開
  - 他拠点からの開発依頼（打ち合わせ・開発）
- 把握方法: **定例ミーティング + Teams + 個別声かけ（口頭のみ）**。
  - 1on1 は**稀（2ヶ月に1回程度）**。
  - 構造化された記録（Issue/進捗表/日報）では見ていない。
- 困りごと（ほぼ全項目該当）:
  - その都度聞かないと分からない / 遅れの発覚が遅い / 全体像が頭に入らない / 属人化 / 把握に時間を取られる。
- 方向性の示し方:
  - 年度・四半期目標は明文化済み。
  - だが**日々の判断軸は頭の中にあるだけで伝えきれていない** / 示す時間が取れていない。
- 遅れ・つまずきの伝わり方: **聞けば答えるが自発的には上がってこない（プル型）**。
- タスク管理ツール: Teams / Gitea / Excel / **自作 TaskFlow（MCP対応済）**。
  - TaskFlow への入力は**週1定例の直前に思い出しながら**（薄く・遅い）。
  - 足りない点: 俯瞰ビューがない / 業務報告が活用できていない / 負荷バランスが見えない。

### 半年後の理想（チーム）= 6項目すべて選択
- 状況が一目で分かる / 遅れが自動で早く上がる / 判断軸が言語化され自走 / 負荷バランスが見える / 業務報告が活用される / 自分のプレイヤー時間が確保。

## 4. 保有する自作システム（6つ）

### 業務・案件の世界
- **ヘルプデスクアプリ**: 他部署からの依頼の入口。
- **TaskFlow**: タスク・担当者・業務報告を管理。MCP対応済。議事録を読み込んで担当割り振る機能あり（ただしアップロード手動）。複数の生産技術部署で稼働中。追加要望が多い。
- **Gijiraku**: 会議から議事録を自動生成。

### 設備の世界
- **Axis**: 設備単位で図面・摘要表・ドキュメントを管理。**作成中・モデルラインでテスト段階**。バグ・新機能要望が多い。
- **YAGI**: 設備のトラブル・メンテ記録ツール。
- **QUICK**: 稼働分析ダッシュボード（トラブル分析・機種段替え分析・品質トレンド）。新機能要望が多い。

### 現状の重要事実
- **Claude 環境はチーム全員に配布済み**。これで業務効率化・チームマネジメントできないか検討中。
- ただし**コーディングする人以外は Claude を業務で使っていない**（使う必然性・きっかけがない）。

## 5. 欲しい「配線」（未接続の連携・5本）

1. **ヘルプデスク → TaskFlow**: 依頼が来たら自動でタスク化+担当割当（今はTeams個別対応でクローズしてしまう）。
2. **Gijiraku → TaskFlow**: 議事録のアクションアイテムを自動でタスク化（今は手動アップロード）。
3. **YAGI → Axis**: メンテで改良（例: ハンド改良）したら図面更新が必要 → Axis に反映。
4. **QUICK → 通知/エスカレ**: 設備トラブルで工作技術に通知、30分経過で別メンバーへエスカレーション（未実装）。
5. **設備トラブル（ドカ停） → YAGI**: ドカ停したら記録を残す連携。

## 6. 診断（このセッションの中核分析）

### 根本問題（ROOT CAUSE）
> イベントが起きた時（依頼が来た/会議が終わった/設備が止まった/改良した）に、
> 情報が「薄く・遅く・口頭で」しか残らない。
> だから俯瞰できず、振れず、自走できない。すべてが本人の頭と問いかけ頻度に依存する。

### 同じ病が3層で反復
- **個人**: 3プロダクトの要望を一人で捌く。要件を固めず着手→手戻り→仕様が頭の中→振れない。
- **チーム**: 口頭把握・プル型。TaskFlowはあるが入力が薄い・遅い。
- **設備**: 器（Axis/YAGI/QUICK）はあるがツール連携がなく、事象が記録されず消える。

### 個人の評価（多角的・本人「おおむね合っている」と承認）
**強み**:
- 課題を見つけ、動くプロダクトを自作し、複数部署・他拠点に展開してきた実行力。
- 自分の限界に自分で気づける内省力。

**問題点5つ**:
1. **単一障害点（SPOF）**: 開発・保守・判断・把握がすべて本人に集約。倒れたら複数部署が止まる組織リスク。
2. **プレイヤー/マネージャーの綱引きに負けている**: コーディング6割。「自分でやったほうが早い」で委譲が進まず、悪循環。技術者出身マネージャーの典型的罠。
3. **対症療法（リアクティブ）**: 来たものに反応するだけで、来る前に設計できていない。アプリが増え保守が積み上がる。優先順位基準が言語化されていない。
4. **記録の揮発**: 要件も進捗も設備事象も口頭で消える。すべての問題の土壌。
5. **自己負荷が不可視**: 報告していないため、抱えている量が組織に見えない。増員交渉も自己防衛もできない。

**総括**: 「優秀さの代償」。何でも自分でできるがゆえにすべてを集約させ、組織リスクと自己消耗を生んでいる。本人は既に気づき始め、解決の器も持っている。

## 7. 課題リスト（整理済み）

### 個人の課題
1. **自分を単一障害点から外す（最優先）**: 頭の中の3つ（仕様・優先度判断軸・方向づけ）を外に出す。
2. **プレイヤー→マネージャーへ重心移動**: コーディング比率を意図的に下げ、遅くても委譲する。
3. **リアクティブ→プロアクティブ**: 優先順位基準の言語化、アプリ化の判断基準を持つ。
4. **自分の負荷を可視化**: 自己防衛と交渉材料。

### チームの課題
1. **記録が揮発する文化→貯まる仕組み（最優先）**: 配線で「自然に貯まる」形へ。
2. **プル型→プッシュ型の把握**: 状況が自動で上がる/俯瞰できる。
3. **メンバーの自走**: 判断軸を言語化・共有。
4. **全員がAIをパートナーに（横断的前提）**: 1〜3の前提条件。
5. **アプリ間の配線**: 設備・案件の世界での「記録が貯まる」の実装。

### 個人とチームは鏡像
- 本人が「頭の中を外に出す」=チームの「記録が貯まる」
- 本人が「委譲する」=チームが「自走する」
- 本人が「設計する」=チームの「プッシュ型把握」
→ 本人が変わることとチームが変わることは同一プロジェクト。

### 着手すべき順序（重要）
1. 記録が貯まる仕組み（土台）
2. 全員がAIを使う（1の前提条件＝実質最初）
3. 配線（1を加速）
4. その上で俯瞰・自走・委譲
→ **1と2を飛ばして委譲・俯瞰をやっても失敗する。**

## 8. AI活用の根本戦略（このセッションで決定）

### ビジョン
- **「アプリを増やす」から「AIを仕事のパートナーにする」へ**。
- アプリを増やすほど保守対象・属人化が増え、本人の首を絞める（課題3そのもの）。

### アプリ化の判断基準（決定した物差し）
> **「貯める・見せる・自動で確実に回す」はアプリ。
> 「考える・整理する・作る・判断する」は純正Claude+スキル。**

- 既存アプリ（TaskFlow=貯める / QUICK=見せる / Axis=貯める）は**この基準で正しい**。
- 今後「報告作成アプリ」「要件整理アプリ」等を作るのは**やめるべき**（純正Claude+スキルの領域）。

### 転換の方向
- 新しいアプリは増やさない。
- 既存の器（TaskFlow等）に純正Claudeを MCP で繋ぐ。
- 「考える・整理する」は純正Claude+スキルが担う。
- 器と器の間の「確実に回す」部分だけ配線（連携）を作る。

### TaskFlow 全員MCP接続の位置づけ
- 単なる俯瞰の道具ではなく、**全員がClaudeを使い始める「きっかけ装置」**。
- 全員が通る動線（TaskFlow）上にClaudeを置く＝必然性が生まれる。
- ただし入力の質がボトルネック。接続と同時に「入力が自然に増える仕組み（配線）」が必要。

### 「記録が自然に貯まる」の実現条件
- 理想は入力レス（思い出し入力をやめ、作業の実体から生成）。
- Claude上で作業していれば、対話の実体から正確な記録を生成できる（入力でなく生成）。
- **前提条件: 全員がClaudeを業務で使っていること**。
  → 「AIをパートナーに」はふわっとした理想でなく、TaskFlow活用の技術的前提。

### 「使っていない人」を巻き込む打ち手
- 号令では無駄（意欲でなく入り口がない構造の問題）。
- **業務動線上にClaudeを置く**。最初のきっかけは1個に絞る。
- 最有力候補=**業務報告/日報の自動生成**（全員がやる必然性 × 全員が面倒 × TaskFlowに記録が貯まる）。
- メンバーの動機（楽したい）と本人の目的（記録を貯めたい）が同一行動で両立。

## 9. 開発・デプロイ基盤（現状と決定）

### 現状
- データソース/ソースは **Gitea**。
- 開発フロー: Gitea からクローン → 開発 → プルリクエスト。**この基本構造は正しい**。
- アプリは **ローカルWeb系** と **AWS系** の2種類。

### 指摘した改善余地（優先順）
1. **docs as code（最優先）**: ソースと同じリポジトリの /docs に要件・設計・ADR を Markdown で置き、コードと一緒にバージョン管理。`engineering:architecture` `engineering:documentation` スキルで開発しながらドキュメントが副産物として生まれる。→ 属人化に直接効く。
2. **マージ前のAIコードレビュー**: 一人運用でプルリクの第三者レビューが形骸化。`tier-a-skills:code-review` `engineering:code-review` をマージ前に通す。
3. **Gitea Actions で CI/CD**: テスト・ビルド・デプロイの自動化。
4. **Gitea MCP連携**: Issue→ブランチ→実装→プルリクを半自動化。保守ループを軽くする。

### 注意
- これら自体が新しい保守対象になる。一度に全部やると課題3が再発。
- **1プロダクト（テスト段階のAxisが実験台に最適）で1つずつ試す**。

### デプロイ品質ゲート（決定・成果物化済み）
- リリース判断を「記憶・注意力」から「固定された自動ゲート」へ移す。
- **AWS系とローカルWeb系で基準を分ける**（脅威モデルが違う。混ぜない）。
- AWS系=フル装備（セキュリティゲート必須）から着手すると決定。

#### 7ゲート構成（AWS系）
1. 静的チェック（Lint/型/**シークレット混入検出**/依存脆弱性/AIレビュー）
2. テスト（ユニット、カバレッジは「下げない」基準）
3. ビルド（本番ビルド+サイズ異常検知）
4. Playwright E2E（**クリティカルパスのみ**。@critical タグ。網羅しない）
5. セキュリティ（checkov IaCスキャン + **SecurityHub CRITICAL 抵触で即停止**。HIGHは運用ポリシー次第）
6. デプロイ（ロールバック用に現行バージョン記録）
7. 事後確認（ヘルスチェック+本番スモーク+SecurityHub再取得）

#### E2Eで確認する項目の原則
- 「壊れたら誰かが困る経路」だけ。ログイン〜主要画面、中核機能1〜3個、書き込み操作。
- 見た目・めったに使わない画面は入れない（保守コストが見合わない）。

#### セキュリティ判断基準（明文化）
- CRITICAL=デプロイ中止 / HIGH=警告+続行可否確認 / MEDIUM以下=記録のみ。
- レベル不問で即停止: シークレット混入 / 重要ポートの0.0.0.0/0公開 / S3意図しないpublic化。

#### 進め方（重要）
- 1プロダクトで完成 → テンプレ化 → 他はコピー+E2E項目差し替え。毎回ゼロから書かない。

## 11. ncr-quality-assurance スキルの理解（深掘り済み）

- 不適合報告を「記録→分類→原因究明→再発防止→月次集計」まで一気通貫支援。
- 3階層分類: 検出ポイント × 不適合カテゴリ（6種）× 4M（Man/Machine/Material/Method）。
- 8D（D1〜D8）テンプレ。D4で5 Whys 強制。
- FMEAは旧RPN方式（automotive-suite の pfmea-builder は AIAG-VDA 2019 AP方式。役割分担すべき）。
- 品質ループ: NCR→8D→lessons-learned（教訓台帳）→PFMEA→Control Plan/SPC→現場へ。
- 8Dと教訓台帳は別物（8D=解決の証拠/全保存、教訓台帳=蒸留した索引/次に引く）。

### 16-1. 8D と教訓台帳の役割分担と具体例（§11補完）

**8Dと教訓台帳は別物（両方残す。捨てない）**
| | 8Dレポート | 教訓台帳(lessons-learned) |
|---|---|---|
| 何の文書か | 1件の不良の解決記録 | 多くの案件から抽出した学びの一覧 |
| 形 | 1案件=1枚の詳しいレポート | 1学び=台帳の1行 |
| 中身 | D1〜D8全工程、固有名詞だらけ | 「教訓」と「次への行動」に凝縮 |
| いつ使う | その不良を解決するとき | 次の新しい仕事を始めるとき |
| たとえ | 1試合の詳しい試合レポート | 部活の引き継ぎノート |

8Dだけではダメな理由: ①量が多すぎて読めない（3年で300件なら300枚、誰も読まない＝死蔵）②固有名詞に学びが埋もれる ③検索・分類できない。
教訓台帳の役割: 固有名詞をそぎ落とし、繰り返し使える学びだけを1行に「蒸留」する。
別スキルにする理由: 蒸留を人手でやると必ず形骸化する（8Dを書いて力尽き転記放置）。自動で8D出力から要点抽出→台帳1行に変換させ「書いたのに死蔵」を防ぐ。

**具体例（3年分の3つの8D → 蒸留 → 先回り）**
- 2024/03 NCR-0042 ドリル加工: 穴径規格外で顧客クレーム3個。真因=手順書にドリル刃の交換基準なし。対策=「500個で刃交換」追記。
- 2024/09 NCR-0088 プレス加工: 製品に傷。真因=金型の摩耗チェック基準が手順書になし。対策=点検周期追記。
- 2025/06 NCR-0151 切削加工: 寸法ばらつき。真因=バイト交換基準が手順書になし。対策=交換基準追記。
→ 教訓台帳に蒸留すると3行が全て同じことを言っている＝**上位教訓「消耗する工具を使う工程は、手順書に交換・点検基準が必要。これは工程横断の弱点」**を発見。
→ 2026/04 新製品（研削＝過去トラゼロ）のPFMEA作成時、「機械加工/Method」で台帳を引くと上位教訓がヒット。砥石も消耗するため、**一度も不良を出す前に**「砥石摩耗による寸法不良／砥石交換基準を手順書明記」と先回り対策できる。
→ 8Dだけなら13枚を誰も読まず、研削でも同じ「工具交換基準ぬけ」で4件目を出す。教訓台帳があるから3件の失敗を研削で繰り返さずに済む。
要点: 8D=起きた1件を解決した証拠（全保存）、教訓台帳=複数の8Dを蒸留し次に先回りで使える知恵（索引）。この2段構えで「過去トラを反映したPFMEA」が仕組みとして実現する。

### 16-2. PFMEAに過去トラを反映する3つの方式（設計案）
汎用 pfmea-builder はJSON入力＋上流成果物しか食わず、自社の過去トラ記憶を持たない。過去トラを構造化入力として注入する現実的方式:

1. **Foundation FMEAライブラリを共有層に置く**: 設備種別×工程×故障モードのベースラインをGiteaにxlsx/構造化データで保持。新規PFMEA = ベースライン + 差分。「チーム共有 vs 個人私有レイヤ」の共有層コンテンツ。AIAG-VDA 2019のFoundation/Family FMEA概念とも合致（白紙から作るな、土台から差分で作れ）。
2. **過去トラ抽出の前段スキルを噛ませる**: pfmea-builderの前に「指定設備クラスのNCR/8D/不具合報告を検索し候補故障モード一覧を出す」スキルを1段置き、その出力をJSON入力に渡す。xlsxチェーン設計に素直に乗る。
3. **発生度(Occurrence)を実データで埋める（最も効く）**: Occurrence評価を勘でなく過去ライン/市場の実際の発生頻度で裏付ける。PFMEAの一番あやしい列が履歴ベースの根拠付きになる。

ループの部品は既に揃っている: 手元の ncr-quality-assurance ＋ automotive-suiteの 8d-problem-solving-builder / lessons-learned-builder を繋ぐと「8Dでクローズ→教訓台帳に蓄積→次のPFMEA入力」のヨコテン循環が回る。
前提（重要）: 過去トラは自由記述・設備名表記バラバラのため、そのままではPFMEA列（工程ステップ/4M/故障モード語彙）に写像できない。**故障モードの統制語彙と設備分類タクソノミーを先に定義**することが土台（「競争優位はデータ蓄積よりオントロジー/意味定義に移る」と整合）。
進め方: 過去トラが豊富な1設備クラスに絞り、②の前段スキルを試作→pfmea-builderに流す、の小さいチェーンで検証。

### 16-3. 製品品質NG と 設備異常 の2ループ並走構造
2つの異常は性質が違う。同じループに混ぜると壊れる。

| | 製品品質NG | 設備異常 |
|---|---|---|
| 困りごと | できた物がダメ | 作る機械がダメ |
| 例 | 穴径外れ・傷・寸法ばらつき | 軸受異音・過熱・チョコ停・突発停止 |
| 失う物 | 品質・信用 | 稼働時間・生産量 |
| 主な手法 | NCR→8D→PFMEA | 故障記録→RCA→RCM/保全計画 |
| 規格 | IATF 16949 | RCM・信頼性工学 |

ただし両者は根でつながる: 設備異常がしばしば製品NGの原因（例: ドリル刃の摩耗→穴径NG）。設備劣化→やがて製品NGとして表面化。

```
【製品品質ループ】 不良 → ncr-quality-assurance(分類・8D) → 真因究明
                                                  ↓
【設備異常ループ】                            ★ lessons-learned(教訓台帳)＝合流点
故障 → 故障記録 → RCA(根本原因) → RCM ↗          ↓
                                          PFMEA / 保全計画に反映
```
教訓台帳が2ループの合流点。「ドリル摩耗で穴径NG」は製品品質の学びかつ設備保全の学び。台帳に「工具の交換基準」として残せば、次の製品設計PFMEAにも次の設備保全計画にも効く（1教訓が両方に活きる）。

FMEAも2種類: PFMEA(工程FMEA、製品品質側)／設備FMEA・FMECA(設備異常側、RCM入口)。automotive-suiteはPFMEAに強く、robreliabilityは設備FMEA・RCMに強い。両方そろえて初めて製品と設備の両輪をカバー。

ユーザーの現状: 保有tier-aは製品品質側（ncr-quality-assurance）に寄り、設備異常側は手薄（structural-analysisは設計検証用で日常の設備故障対応とは別物）。→ robreliabilityの信頼性スキルを設備側入口として追加するのが筋（§15-1で4スキルをCowork用にパッケージ済）。検証する設備クラスは、製品NGと設備異常の両方が起きる工程を選ぶと2ループ連携まで一度に検証できる。

### 16-4. 「高校生でもわかる」説明で用いた比喩（参考・任意）
- ncr-quality-assurance = 工場の失敗ノート係（保健室の記録ノートの工場版）。記録→本当の原因を5回掘る→二度と同じ失敗をさせない。
- 4M = 人(Man)/機械(Machine)/材料(Material)/やり方(Method)、全部Mで始まる合言葉。
- 5 Whys = 「作業者がミスった」で止めず5回なぜを掘ると「人のせい」でなく「仕組みのせい」に行き着く。犯人探しでなく原因つぶし。
- 暫定対策 vs 恒久対策 = 火に水をかけて消す(暫定) vs 火が出ない仕組みにする(恒久)。混ぜない。
- 8D = 不良を「応急処置→真因究明→恒久対策→横展開」までやり切る世界標準の8段階フォーマット。各ステップは「ここでサボると失敗する」関所。
- 教訓台帳 = 部活の引き継ぎノート（代が替わっても同じ失敗をしないよう書き残す）。

## 17. 到達目標：チーム・コックピット構想（このセッション終盤で確定）

### 17-1. 何を作りたいか（ユーザーのビジョン）
Claudeを有効活用し、**チームの司令塔（コックピット）**を実現する。具体的に見えるようにしたいもの:
- 各メンバーの年間目標が入っていて、進捗が目標通りか
- 方向性がずれていないか
- 新しい目標が生まれたら拾えるか
- 各ツール（Axis/QUICK/TaskFlow等）がどう連携しているか
- 各メンバーがどう連携して働いているか
- 仕事の負荷が平滑化できているか
- これら全体像が一目で分かること

位置づけ: これは §6〜8 で見えた課題（単一障害点・プル型把握・俯瞰がない）への、ユーザー自身が出した答え。方向性は正しい。

### 17-2. 最重要原則：これを「アプリ」にしない
今日決めた判断基準（§8）を適用する。
> 「貯める・見せる・自動で確実に回す」はアプリ。「考える・整理する・判断する」は純正Claude+スキル。

コックピットの中身は大半が「考える・判断する」側:
- 「目標通りか」＝進捗と目標を照合して**判断**
- 「方向性がずれていないか」＝活動を目標と突き合わせて**評価**
- 「負荷が平滑化できているか」＝タスク分布を見て**分析**
- 「全体像」＝散らばった情報を**統合して語る**

→ これらは固定ダッシュボードが最も苦手。ダッシュボードは数字を見せられるが「ずれている」「この人に偏っている、◯◯さんに振り直すべき」という判断・提案はできない。作っても結局あなたが数字を見て判断＝今と変わらず、保守対象だけ増える（単一障害点を悪化）。
**結論: コックピットをダッシュボードアプリとして新規開発してはいけない。**

### 17-3. 正解：コックピットは「画面」でなく「問いかけ」で実現する
データは既存アプリ（TaskFlow等）に置いたまま、ClaudeにMCPで読ませ、ユーザーが問いかける。

```
データの置き場所（新規開発なし・既存アプリ活用）
  - 年間目標 → TaskFlow か、目標専用の軽い置き場（置き場所の決定は保留事項）
  - 進捗・タスク・担当 → TaskFlow
  - ツール連携状況 → 各アプリ
        ↓ MCPで接続
頭脳（純正Claude + スキル）
        ↓ ユーザーが問いかける
  例:「今週、目標に対して遅れているメンバーと案件を出して」
    「各メンバーの負荷を一覧にして、偏っている人と余裕のある人を教えて」
    「先月から、当初の年間目標になかった新しい仕事が増えた人は誰？」
    「方向性が目標とずれ始めている兆候があれば指摘して」
```
コックピット＝「画面」ではなく「問いかけたら答えが返る状態」として実現する。新アプリ不要、今日の原則のまま欲しいもの全部が手に入る。

### 17-4. 成立条件（＝今日の積み上げそのもの）
このコックピットが機能する前提:
1. データが貯まっていること（目標も進捗も構造化されて存在）＝ §7チーム課題1
2. 全員がClaudeで働いていること（でないと活動実体が記録に残らない）＝ §8 / §7チーム課題4
3. TaskFlow等がMCPで繋がっていること（Claudeがデータを読める）＝ §8・保留中

→ **コックピットは最後に乗るもの。いきなりは作れない。** §7の着手順序で言えば「記録が貯まる→全員が使う→繋がる→俯瞰」の最終段。これは**ロードマップ Phase 2 の完成形**にあたる。

### 17-5. 慎重に扱う論点：監視でなく采配の道具に
「負荷の見える化」「目標進捗の管理」は強力だが諸刃。
- 誤ると、メンバーに「監視されている」と映り、TaskFlow入力が防御的になり、正確な記録がかえって集まらなくなる（土台が崩れる）。
- 日報スキルで「査定しない・本人を責めない」とガードレールを入れたのはこの予防。
- **司令塔はユーザーが采配するための道具であって、メンバーを評価する道具ではない。** この線引きを技術より先に、設計思想として明確にする。

### 17-6. このビジョンが持つ意味
コックピット構想は、今日バラバラに扱った全テーマ（属人化解消・記録を貯める・全員AI活用・配線・俯瞰・委譲）の**収束点**であり、「何のために全部やるのか」の答え。
ロードマップ上の到達目標として固定する。実現の最初の一歩は §13 と同じ＝TaskFlow-MCP接続と、年間目標データの置き場所の決定。

## 18. 構想：Issue起点のAI開発ループ（自動化）

### 18-1. やりたいこと（ユーザーのビジョン）
Issueを起点に、AIが開発ループを回す。2モード:
- **モードA（全自動）**: `AI`タグ付きIssue → 毎日自動でAIが実装 → プルリク → デプロイまで無人。
- **モードB（半自動・人間ループ）**: タグなしIssue → AIがやってみる → 判断が要る所はIssueコメントに質問を書いて**停止** → ユーザーがコメントで返信 → その先へ進む。

位置づけ: 「保守がしんどい（量×重さ×属人化）」への最も直接的な打ち手。aws-deploy-gate（§9）の上流（Issue→実装）を自動化する話で、ゲートが先にあるからAIのプルリクも安全に流せる。

### 18-2. 技術構成（課金を踏まえた推奨構成。詳細は §18-8）
**推奨: Cowork の Scheduled Tasks を入口にする（サブスク内で回すため）。**
1. **トリガー**: Cowork のスケジュールタスクで毎日起動、`AI`ラベルのOpen Issueを拾う。
2. **エージェント**: **Cowork自身が** Giteaをclone→Issue本文を読んで実装→コミット→プルリク作成。**`claude -p`（ヘッドレス）は使わない**（API課金になるため。§18-8参照）。
3. **対話チャネル**: Issueのコメント欄を、AIと人間の往復の場に使う（AIが質問を書く／人が答える／AIが再開）。
※ 前回検討した「Gitea Actions cron + claude -p」は **API課金になるため不採用**（§18-8）。
※ ただしチーム規模の本番自動化では、サブスク内に粘らず Claude Platform（APIキー従量）で設計するほうが健全な場合がある（§18-8 留意1）。
※ Coworkスケジュールで完全な実装ループが安定して回るかは要実機検証（§18-8 留意2）。

### 18-3. 最重要原則：最初から全自動デプロイを目指さない
全自動デプロイ＝AIが誤コードを誰のレビューも経ず本番に出せる＝複数部署が使うプロダクトで最大の事故源。
**信頼を積みながら自動化範囲を広げる4段階:**
1. **AIはプルリクまで。デプロイは人が承認**（AIのコードの信頼度を体感。aws-deploy-gateの7ゲートをAIのプルリクも通す）
2. **モードBの人間ループを作り込む**（最も価値が高く、最も難しい）
3. **低リスクなものだけ全自動を解禁**（ローカルWeb系の・小さな・後戻りできる変更のみ。AWS系/破壊的変更は人承認維持）
4. **全自動範囲を慎重に拡大**（AWS系本番デプロイは最後まで人承認を残すのが賢明）

### 18-4. 設計の核心：「AIが"聞くべき時"を判断できるか」
モードBの成否はここ。何でも聞けば使えない、聞かず突き進めば事故る。判断基準を明文化したスキルが必要。
- **止まって聞くべき**: 要件が複数解釈できる／データ破壊（削除・スキーマ変更・マイグレーション）／外部影響（他ツール連携・API仕様変更）／セキュリティ判断。
- **聞かず進んでよい**: 明確なバグ修正（再現手順あり・期待動作が一意）／表示の微修正・文言変更／テストで正しさが自動検証できる範囲。
→ これは §2 の「対応すべきか毎回自分で判断している」判断軸の言語化そのもの。属人化解消に直結。

### 18-5. 進化形：過去の判断から学ぶ「判断学習エージェント」（ユーザー追加構想）
固定ルール型でなく、**ユーザーとAIの過去の判断の蓄積から学んで判断**させたい。判断基準自体が使うほど賢くなる。
- 固定ルール型: 人が書いた静的ルール（「データ破壊は聞く」）
- **事例参照型**: 「過去に似たIssueでユーザーはこう判断した→今回もこうする」。人間の熟練（規則集の新人でなく経験のベテラン）と同じ。

**材料は既にGiteaに存在する（新規作成不要・判断の化石）:**
- Issue履歴（どんな要望に、どういう判断で実装/却下/保留したか）
- Issueコメント（どこで「こう実装して」と指示したか）
- プルリク履歴（何を承認し、何を差し戻したか）
- コミットメッセージ（なぜその変更をしたか）

**仕組み（自己強化ループ）:**
```
新Issue → 過去の類似Issue/PRを検索（Gitea） → 「過去にどう判断したか」を参照
   → 進む or 聞く を決める → 判断結果と理由を記録（次の判断材料になる）
```
AIの判断結果自体が次の判断材料として蓄積され、使うほど洗練される。

### 18-6. 判断学習の設計上の注意（4論点）
1. **過去の判断が正しいとは限らない**: 間違い・例外も混じる。参照は「傾向」であって「絶対の正解」ではない。「過去はこうだが今回は事情が違うかも」と問える余地を残す。
2. **判断の"理由"が記録されていないと学べない**: 結果だけでは学べない。「なぜ実装/差し戻したか」が要る。→ **【現状の制約】ユーザーのIssue/PRは「結果は残るが理由は薄い（やった事実中心）」。** よってこの仕組みは、理由を残す運用を整えてからでないと機能しない。
3. **類似ケースの判定が難しい**: キーワード一致では取り違える。Issueの分類軸（機能追加/バグ/データ変更/外部連携…）＝統制語彙・オントロジーが整うと精度が上がる。
4. **作り込みすぎない**: 機械学習基盤として作ると課題3に逆行。実態は「Claudeに過去のIssue/PRを読ませ『過去を踏まえ今回どうすべきか』と問う」だけで大半が実現。純正Claude+MCP(Gitea連携)で足りる。新アプリではない。

### 18-7. 暴走・費用の安全弁（必須）
- 1Issueあたりの試行回数・時間に上限（無限修正の防止）
- AI生成プルリクには必ず「AI生成」ラベル（人が識別できる）
- 想定外に大きな差分（大量ファイル変更）は自動停止して人を呼ぶ
- **API利用料の日次上限**（毎日全Issue処理はコストが積む。下記課金論点参照）

### 18-8. 課金の論点（重要・2026-06-15の課金改定を反映して更新）

**【最重要・禁止事項】無人自動化で `claude -p`（ヘッドレスのClaude Code）を使わないこと。**
`claude -p` は対話的利用ではなくプログラム的実行に分類され、サブスク枠でなく **API従量課金（別クレジット）** になる。cron や Cowork から `claude -p --dangerously-skip-permissions`（yoloエイリアス）を呼ぶ構成は、知らぬ間に高額API課金を生む（実例: Max契約者が `claude -p`＋APIキーのスケジュール実行で2日間$1,800超の課金）。**自動化スクリプトで `claude -p` を呼ばない。シェルのANTHROPIC_API_KEYを継承させない。**

**2026-06-15 課金改定の構造（2プール制）:**
- **インタラクティブ・プール（サブスク内・従来通り）**: Web/デスクトップ/モバイルのチャット、ターミナルで対話利用するClaude Code、**Claude Cowork（スケジュールタスク含む）**。
- **Agent SDK クレジット・プール（API料金・別枠）**: Claude Agent SDK、`claude -p`（ヘッドレス）、**Claude CodeのGitea/GitHub Actions（CI起動）**、Agent SDK経由で認証するサードパーティアプリ。月次クレジット（Pro $20 / Max5x $100 / Max20x $200相当）を消費し、使い切ると停止（繰越なし、overflow有効時のみAPI従量で継続）。

**構成ごとの課金（決定的な分かれ目＝「誰が実際にコードを書くか」）:**
| 構成 | 課金 |
|---|---|
| Cowork の Scheduled Tasks 内で、**Cowork自身が** clone・実装・プルリクまでやる | **サブスク内** |
| Cowork や cron から **`claude -p`** を呼び出す | **API課金（禁止）** |
| **Gitea Actions（CI）** から Claude Code を起動 | **API課金** |

→ **推奨構成: 「Coworkのスケジュールタスクで、Cowork自身がGiteaをclone→実装→プルリク」**。`claude -p` を介さない限りサブスク内で回せる。前回案（Gitea Actions cron + claude -p）は **API課金になるため不採用**。

**ただし2つの留意:**
1. **規模**: 公式は「Agent SDKの月次クレジットは個人の実験・自動化向けサイズ。チーム共有の本番自動化はClaude Platform（APIキー）の予測可能な従量課金にすべき」と明記。本構想（毎日・全Issue・複数プロダクト・チーム運用）は本番自動化に近く、サブスク内で粘るより最初からAPI課金（Claude Platform）で設計するほうが健全な可能性がある。コスト試算のうえ判断する。
2. **機能の実現性（課金とは別問題）**: Coworkのスケジュールタスクで、Git操作を含む完全な実装ループ（clone→変更→テスト→プルリク）が安定して回るかは、実際に試して確認が必要。

**※課金体系・機能仕様は変動が激しい。実装前に必ず公式ヘルプセンター（support.claude.com）で最新条件を確認すること。特に自分のプラン種別での扱いは契約画面で確認する。**


### 18-9. 今日の全構想との一貫性
本構想（過去のIssue/PRをClaudeに読ませ判断を引き出す）は、コックピット構想（§17、TaskFlowをClaudeに読ませ俯瞰を引き出す）と同じ構造。
共通原則: **新アプリを作らず、既存の記録をClaudeに読ませて知性を引き出す**（§8の判断基準の応用）。一貫している。

### 18-10. ロードマップ上の位置づけと着手順序
- 段階1（AIはプルリクまで・人が承認）は、aws-deploy-gate 完成後に着手可能。比較的早く始められる。
- 判断学習（§18-5）は、その前提として「**Issue/PRに判断理由を残す運用**」の確立が必要（現状は理由が薄いため）。これが最初の地味だが重要な一歩。
- 推奨順: ①aws-deploy-gate完成 → ②段階1（AI実装→PR、人が承認）→ ③判断理由を残す運用を整える → ④モードBの判断スキル → ⑤判断学習（過去事例参照）→ ⑥低リスクのみ全自動。
