Internal Diagnosis · 30-question synthesis

業務診断と全体ロードマップ

個人業務(前半20問)＋チームマネジメント(後半10問)＋保有システムの統合分析。
ソ企／生産技術DXの現状を1枚に固定し、最初の一手を決めるための見取り図。
01 · ROOT

すべての課題は、ひとつの根本問題の現れ

バラバラに見えた悩みを30問でたどると、源は一点に収束する。

ROOT CAUSE

イベントが起きた時(依頼が来た／会議が終わった／設備が止まった／改良した)に、
情報が 薄く・遅く・口頭で しか残らない。だから俯瞰できず、振れず、自走できない。

結果、すべてが「あなたの頭の中」と「あなたが聞きに行く頻度」に依存する。個人もチームも設備も、同じ構造。

02 · LAYERS

同じ病が、3つの層で繰り返されている

個人 → チーム → 設備。現れ方は違うが、原因は同一。

LAYER 1個人業務
コーディング6割のプレイングマネージャー。Axis・QUICK・TaskFlow という3つの自作プロダクトに複数部署から要望が殺到し、一人で捌いている。要件を固めず着手 → 手戻り → 仕様が頭の中だけ → 誰にも振れない。
保守が重い属人化文書が生まれない振れない
LAYER 2チームマネジメント(6〜10名)
担当はバラバラ(開発・教育・ヘルプデスク・インフラ調整・社外展開・他拠点)。把握は会議とTeamsの口頭のみ、1on1は2ヶ月に1回。遅れは聞かないと上がってこない(プル型)。TaskFlowはあるが、入力は週1定例の直前に思い出しながら。
全体像が掴めない遅れの発覚が遅い負荷バランス不明業務報告が活かせない
LAYER 3設備・現場
Axis・YAGI・QUICK という器はある。だがトラブル・改良・メンテのイベントが、別ツールへ自動で流れない。設備が止まってもTeamsで個別対応、記録が残らずクローズされる。
連携なし通知・エスカレなし図面更新の取りこぼし
03 · ASSETS

部品は揃っている。足りないのは「配線」だけ

あなたは既に6つの自作ツールを持つ。課題は新規開発ではなく、ツール間の接続。

業務・案件の世界

ヘルプデスク他部署からの依頼の入口
TaskFlowタスク・担当・業務報告／MCP対応済／議事録取込み機能あり
Gijiraku会議から議事録を自動生成

設備の世界

Axis設備単位で図面・摘要表・文書を管理／テスト段階
YAGI設備トラブル・メンテ記録
QUICK稼働分析ダッシュボード(トラブル／段替え／品質)
欲しい配線 — 未接続(5本)
1ヘルプデスク→TaskFlow依頼が来たら自動でタスク化＋担当割当(今はTeams個別対応でクローズ)
2Gijiraku→TaskFlow議事録のアクションアイテムを自動でタスク化(今は手動アップロード)
3YAGI→Axis改良(例:ハンド改良)で図面更新が必要 → Axisへ反映
4QUICK→通知・エスカレトラブルで工作技術へ通知、30分経過で別メンバーへ
5ドカ停→YAGI設備が止まったら記録を残す連携
稼働中 テスト段階 未接続の配線
04 · GOALS

「全部やりたい」は正しい — 4つは一つの構造の側面

別々の願いではなく、同じ根を断てば連鎖して効く。

要望管理・優先度3プロダクトの洪水を捌く
業務効率化配布済Claude環境の活用
チーム俯瞰・委譲頭の中→記録へ
アプリ間連携イベント駆動の配線

いずれも「記録が自然に生まれ、自動で俯瞰される」状態に到れば同時に前進する。

05 · ROADMAP

順番がある:配線が先、俯瞰は後

構造化データが流れて初めて、product-management や TaskFlow-MCP が効く。

PHASE 0 · NOW
基盤と土台
  • TaskFlow-MCP を自分の Claude に接続
  • 要望を1本、write-spec で要件化して体感
  • 環境・制約(閉域/プロキシ)の確認
PHASE 1
配線(A層)
  • Gijiraku→TaskFlow(既存機能の自動化)
  • ヘルプデスク→TaskFlow
  • QUICK→通知／ドカ停→YAGI
PHASE 2
俯瞰・委譲(B層)
  • TaskFlowデータをClaudeで俯瞰・異常検知
  • 要望の優先度づけを仕組み化
  • 判断軸の言語化 → メンバー自走
PHASE 3
チーム展開
  • ドキュメント生成を開発フローに内蔵
  • スキル／ワークフローをチーム標準化
  • 属人化の解消 → 振れる体制へ
本資料は対話(30問＋システム聞き取り)から生成した現状診断であり、確定計画ではない。各フェーズの実装可否は社内のネットワーク制約・MCP接続可否の確認後に判断する。
